# Impulse — правила проекта

Проект impulse-web — веб-сервис для студентов и команд. Бизнес-домен: гипотезы, проекты, команды.

## Стек

- **Vite + React + TypeScript** — сборка и фреймворк
- **Radix UI Themes** — UI-компоненты и дизайн-токены
- **react-router** (HashRouter) — роутинг
- **Node.js** + **npm** — менеджмент зависимостей

## Структура проекта

```
src/
  assets/        — SVG, изображения
  components/    — переиспользуемые UI-компоненты (Header, ...)
  pages/         — страницы (соответствуют роутам)
  App.tsx        — роутинг
  main.tsx       — точка входа, обёртка в <Theme>
docs/           — аналитика фич и бизнес-логика (читать перед работой над фичей)
.claude/skills/  — скиллы (правила, инструкции)
```

## Правила кода

- **TypeScript strict** — без `any`, без `@ts-ignore` без веской причины
- **Не создавай `index.css`/`App.css`** — используй стили Radix Themes. Если нужны кастомные стили — делай колокейшн (`Component.tsx` + `Component.css`) или inline-style для точечных правок
- **Иконки:** Radix Icons (`@radix-ui/react-icons`) по умолчанию. Кастомные SVG — только когда Radix не подходит (например, логотип или иконки из Figma)
- **Компоненты Radix Themes** — первый выбор. Не собирай кнопки/формы руками
- **Не редактируй файлы `dist/`, `node_modules/`**
- **Роутинг:** `HashRouter` (на случай деплоя без server-side rewrites)

## Конвенции именования

- Компоненты и страницы: `PascalCase.tsx`
- CSS-файлы компонентов: `Component.css` рядом с `Component.tsx`
- Ассеты: `kebab-case.svg` (например, `logo-impulse.svg`)
- Роуты: `kebab-case` (`/ui-kit`, `/dashboard`)

## Работа со спеками (`/docs`)

Перед работой над любой фичей:
1. Проверь, есть ли спека в `/docs` по этой фиче
2. Если есть — прочитай её полностью перед написанием кода
3. Если в спеке есть противоречие с текущим кодом — задай вопрос пользователю, не додумывай

Спеки — источник правды для бизнес-логики. Код подстраивается под спеку, а не наоборот.

### Триггер синхронизации с документацией

Когда пользователь описывает экраны или фичи и использует фразы вроде:
- «смотри, бизнес-логика следующая:»
- «бизнес-логика такая:»
- «логика фичи:»
- «правило работы:»
- или аналогичные формулировки, вводящие описание правил/сценариев

**Обязательный workflow:**
1. **Найди** релевантные разделы в `/docs` (`student-actions.md`, `data-model.md`, `glossary.md`, тематические файлы фич)
2. **Сравни** описанное пользователем с тем, что уже есть в доках
3. **Одно из трёх:**
   - Если бизнес-логика в доках соответствует — используй её и кратко подтверди пользователю, что всё совпадает
   - Если в доках есть противоречие — покажи оба варианта пользователю и уточни, какой актуален. Затем обнови документацию
   - Если это новая логика — добавь описание в подходящий файл `/docs` (или создай новый) перед тем, как писать код/дизайн
4. **Реализуй** экран/фичу только после того, как документация согласована

Цель — дизайн, код и документация всегда в одном источнике правды. Документация ведёт, реализация следует.

## Git и ветки

- Основная ветка: `main`
- Активная рабочая ветка сейчас: `docs`
- Коммиты — по-русски или по-английски, формат "что сделано, зачем"
- Не коммить без явной просьбы пользователя

## Команды

- `npm run dev` — dev-сервер (порт 5176, см. `.claude/launch.json`)
- `npm run build` — проверка TS + прод-сборка
- `npx tsc --noEmit` — только тайпчек
